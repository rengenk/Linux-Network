   * 194.34.23.100 - нет
   * 127.0.0.2 - да
   * 127.1.0.1 - да
   * 128.0.0.1 - нет

## Part 4. Сетевой экран

#### 4.1. Утилита **iptables**

##### Создать файл */etc/firewall.sh*, имитирующий фаерволл, на ws1 и ws2:
```shell
#!/bin/sh

# Удаление всех правил в таблице "filter" (по-умолчанию).
iptables –F
iptables -X
```
##### Нужно добавить в файл подряд следующие правила:
##### 1) на ws1 применить стратегию когда в начале пишется запрещающее правило, а в конце пишется разрешающее правило (это касается пунктов 4 и 5)
##### 2) на ws2 применить стратегию когда в начале пишется разрешающее правило, а в конце пишется запрещающее правило (это касается пунктов 4 и 5)
##### 3) открыть на машинах доступ для порта 22 (ssh) и порта 80 (http)
##### 4) запретить *echo reply* (машина не должна "пинговаться”, т.е. должна быть блокировка на OUTPUT)
##### 5) разрешить *echo reply* (машина должна "пинговаться")


*  `$ sudo nano /etc/firewall.sh` создаем файл имитирующий фаерволл и добавляем в него правила:   
![](img/4.1.png)
```shell
-F – очистить все правила;
-X – удалить цепочку;
-A – добавить правило в цепочку;
-p – вручную установить протокол (TCP, UDP, UDPLITE, ICMP, ICMPv6, ESP, AH, SCTP, MH);
--dport - указание номера порта
-j – выбрать действие при подтверждении правила.
--icmp-type - указание типа сообщения
```

* Запустить файлы на обеих машинах командами `chmod +x /etc/firewall.sh` и `/etc/firewall.sh`     
* iptables -L - как выполняется просмотр правил
![](img/4.2.png)

Правило `iptables -A OUTPUT -p icmp --icmp-type echo-reply -j DROP` блокирует исходящие ICMP пакеты типа "echo reply". Это означает, что если машина получила пинг (ICMP пакет типа "echo request") от другой машины, она не сможет отправить на него ответный пакет (ICMP пакет типа "echo reply").
Разница между стратегиями, применёнными в первом и втором файлах закобчаеися в том что на ws1 первым указано запрещающее правило на выход, поэтому она не сможет пропинговаться. 
У ws2, наоброт, а значит она сможет пропинговаттьcя.

#### 5.2. Включение переадресации IP-адресов.
* `$ sudo sysctl -w net.ipv4.ip_forward=1` - Для включения переадресации IP, выполните команду на роутерах. При таком подходе переадресация не будет работать после перезагрузки системы.

Флаг "-w" означает "запись" (write). Он указывает утилите sysctl изменить значение параметра с помощью указанного варианта. В данном случае, значение "1" передается в параметр "net.ipv4.ip_forward", чтобы включить переадресацию IP-пакетов.

Параметр "net.ipv4.ip_forward" является параметром в системе Linux, который регулирует переадресацию IP-пакетов между сетевыми интерфейсами. Если значение этого параметра равно "1", то переадресация IP-пакетов включена, и Linux будет пересылать пакеты между различными сетевыми интерфейсами, что позволяет устройствам в разных сетях коммуницировать друг с другом.

![](img/5.4.png)


* Откройте файл */etc/sysctl.conf* и добавьте в него следующую строку:
`net.ipv4.ip_forward = 1` 

  При использовании этого подхода, IP-переадресация включена на постоянной основе.*

  - `"-i enp0s9"` - опция для указания tcpdump о необходимости прослушивать сетевой интерфейс с именем "enp0s8". Сетевые интерфейсы могут иметь различные имена в разных системах.

* В отчёте, опираясь на вывод, полученный из дампа на r1, объяснить принцип работы построения пути при помощи **traceroute**:

  Для определения промежуточных маршрутизаторов traceroute отправляет серию пакетов данных целевому узлу, при этом каждый раз увеличивая на 1 значение поля TTL («время жизни»). Это поле обычно указывает максимальное количество маршрутизаторов, которое может быть пройдено пакетом. Первый пакет отправляется с TTL, равным 1, и поэтому первый же маршрутизатор возвращает обратно сообщение ICMP, указывающее на невозможность доставки данных. Traceroute фиксирует адрес маршрутизатора, а также время между отправкой пакета и получением ответа (эти сведения выводятся на монитор компьютера). Затем traceroute повторяет отправку пакета, но уже с TTL, равным 2, что позволяет первому маршрутизатору пропустить пакет дальше.

  Процесс повторяется до тех пор, пока при определённом значении TTL пакет не достигнет целевого узла. При получении ответа от этого узла процесс трассировки считается завершённым.
